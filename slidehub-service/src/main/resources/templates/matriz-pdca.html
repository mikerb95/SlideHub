FASE PLAN

Plan de Gestión

Basado en ISO/IEC 12207, SlideHub es operado mediante un ciclo iterativo y de entregas (Sprints) en el que se consolidan distintas Fases de diseño y desarrollo, abarcando desde una arquitectura de microservicios distribuida hasta el actual Monolito Modular.

Organización y Roles
  • Requerimientos trazados contra el documento Requerimientos Funcionales y No Funcionales y el CSV maestro de Historias de Usuario.
  • Roles claramente acotados en el sistema: PRESENTER y ADMIN más accesos públicos para dispositivos pasivos (pantallas).
Infraestructura y Base Base de Conocimiento
  • Repositorio controlado con pautas firmes documentadas en AGENTS.md y CLAUDE.md.
  • Pipeline DevOps en Render para control de despliegue unificado (Web Service único: `slidehub-service`).
FASE CHECK

Informe de Verificación y Validación

Evaluamos la calidad operativa y el soporte técnico mediante la evaluación Smoke E2E, los tests automatizados y la revisión sistemática de arquitectura frente a normas internacionales como ISO/IEC 25000 y 27001.

Análisis E2E y Pruebas Unitarias
El proyecto está respaldado por smoke-e2e.sh automatizado en GitHub Actions y el servicio StatusChecksService que expone el estado de salud interno de:
  • DB (PostgreSQL en Aiven), Redis y MongoDB (AI-notes).
  • Servicios externos (Gemini WebClient).
  • S3 Cloud Storage (AWS SDK v2).
Vulnerabilidades Detectadas (Verificación de Seguridad)
ISO/IEC 27001 - Brecha Crítica El persistir Tokens OAuth2 (GitHub/Google) en texto plano (entidad User en BD) ha sido verificado como negativo para el estándar de seguridad. Solución asignada: Cifrado en reposo obligatorio.
FASE ACT & GENERAL

Propuestas del Ciclo PDCA

El Backlog priorizado define cómo la fase *Act* del ciclo PDCA corregirá las brechas encontradas y propondrá mejoras continuas.

PRIORIDAD 0 (Urgente - Remediación Inmediata)
  • Cifrado en reposo de github_access_token y google_refresh_token mediante JPA AttributeConverter. Rotar credenciales actuales.
  • Alinear la documentación técnica real con esta actualización de seguridad.
PRIORIDAD 1 (Corto Plazo - Formalización)
  • Crear plan formal de evaluación de calidad (ISO/IEC 25001) documentando características, métricas y umbrales operables.
  • Validar una plantilla PDCA estándar para los Incidentes de Desarrollo antes de incluirlos en Hotfixes.
  • Establecer KPIs objetivos como *% de éxito Smoke E2E >= 95%*.
PRIORIDAD 2 (Mejora Continua)
  • Autoevaluación de madurez de todos los procesos base alineados a ISO/IEC 15504 (SPICE).
PERSONAL SOFTWARE PROCESS

Iniciativas PSP

Para fortalecer la gestión individual del equipo y la estimación de plazos a lo largo del PDCA, SlideHub incorporará prácticas **PSP (Personal Software Process)** de manera gradual.
  • Introducir una bitácora o log individual de tiempo por historia de usuario para medir desviación de estimación vs tiempo real.
  • Trazar los defectos detectados por fase (codificación, QA/smoke, producción) a la causa raíz particular del desarrollador para generar retrospectivas más asertivas.
  • Objetivo inmediato: Reducir la desviación de esfuerzo al ±25% y garantizar el 100% de cobertura en *Acceptance Criteria* documentados.
